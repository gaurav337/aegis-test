# Aegis-X Pipeline Poor Performance Analysis & Fix Plan

## Problem Summary

The pipeline achieves **44% accuracy** (worse than random 50%) on a 100-image test set:
- **FP=30**: Real images incorrectly predicted as Fake
- **FN=26**: Fake images incorrectly predicted as Real  
- **TN=20**: Real correctly predicted as Real
- **TP=24**: Fake correctly predicted as Fake
- **ROC AUC: 0.4766** — BELOW 0.5, the smoking gun

**AUC < 0.5 means the pipeline has signal but is using it BACKWARDS.** If models were random, AUC would be 0.50. Below 0.5 means the ranking is inverted.

## Root Cause Analysis

### 🔴 CRITICAL BUG #1: Label Inversion — Detectors Are Confident AND Correct, But Pipeline Reads Them Backwards

**Evidence**: REAL images getting extremely HIGH detector scores:
- `real_4743.jpg` (REAL): Xception=0.988 → Pipeline interprets as "98.8% fake" BUT Xception is actually saying "98.8% REAL"
- `real_4367.jpg` (REAL): SBI=0.984 → Pipeline interprets as "98.4% fake" BUT SBI is actually saying "98.4% REAL"
- `real_4120.jpg` (REAL): FreqNet=0.977 → Pipeline interprets as "97.7% fake" BUT FreqNet is actually saying "97.7% REAL"

**These detectors are working correctly** — they're confidently identifying real images. The problem is the pipeline interprets high scores as "high fakeness" when they actually mean "high realness".

**Also explains FN=26**: Fake images get LOW detector scores because detectors correctly see them as fake (low realness). Pipeline interprets low scores as "low fakeness" = REAL. This is why fake_676.jpg (Score=0.6147), fake_2380.jpg (Score=0.6331), fake_5479.jpg (Score=0.6417) are all called REAL — the detectors are correct but the interpretation is inverted.

**Per-Model Analysis**:

#### SBI Tool (`core/tools/sbi_tool.py:499-502`)
```python
# 2-class model: class 0=fake, class 1=real
score_115 = F_sbi.softmax(out_115, dim=1)[0, 0].item()  # Takes class 0
```
**Issue**: Comment says "class 0=fake" but the evidence shows class 0 is actually REAL (high scores on real images). The checkpoint uses class 0=REAL, class 1=FAKE.

**Fix**: Change `[0, 0]` to `[0, 1]` to get the actual fake class:
```python
score_115 = F_sbi.softmax(out_115, dim=1)[0, 1].item()
```

#### Xception Tool (`core/tools/xception_tool.py:273-278`)
```python
def _score_tensor(self, model: nn.Module, tensor: torch.Tensor) -> float:
    """Returns the fake probability."""
    outputs = model(tensor)  # (1, 2)
    probs = F.softmax(outputs, dim=1)
    # Class 1 is usually the fake class in standard FF++ pretraining
    return float(probs[0, 1].item())
```
**Issue**: The comment says "usually" — this checkpoint uses class 0=FAKE, class 1=REAL. So `probs[0, 1]` is actually P(REAL), not P(FAKE).

**Fix**: Change to `probs[0, 0]` or add config:
```python
return float(probs[0, 0].item())  # class 0 is actually FAKE for this checkpoint
```

#### UnivFD Tool (`core/tools/univfd_tool.py:226-228, 230-237`)
```python
logit = wrapper.probe(embeds)
return logit.item()  # Return raw logit, not sigmoid

def _calibrate_score(self, logit: float, temperature: float = 1.0) -> float:
    scaled_logit = logit / temperature
    return float(1.0 / (1.0 + np.exp(-scaled_logit)))  # sigmoid
```
**Issue**: The probe was trained with positive logit = REAL (not fake as assumed). Sigmoid(positive) = high score for real images, but pipeline treats high score as fake.

**Fix**: Negate logit before sigmoid:
```python
def _calibrate_score(self, logit: float, temperature: float = 1.0) -> float:
    scaled_logit = -logit / temperature  # Note the negative sign
    return float(1.0 / (1.0 + np.exp(-scaled_logit)))
```

#### FreqNet Tool (`core/tools/freqnet_tool.py:361-364`)
```python
logit = model(norm_tensor)
neural_score = float(torch.sigmoid(logit).item())
```
**Issue**: Same pattern — model was trained with positive logit = REAL. Sigmoid(positive) = high score for real images.

**Fix**: Negate logit:
```python
neural_score = float(torch.sigmoid(-logit).item())  # Note the negative sign
```

### 🔴 CRITICAL BUG #2: Some Detectors Are Dead/Not Working (Near-Constant Output)

**Evidence**:
- `run_dct`: Every image → 0.03 to 0.15 (near-constant) — **BROKEN**
- `run_univfd`: Every image → 0.30 to 0.49 (near-constant) — **BROKEN** (also label inverted)
- `run_corneal`: Every image → 0.000 — **DEAD**

A working model does NOT output the same value for every input. These are either:
1. Weights not loaded (random initialization produces uniform outputs)
2. Preprocessing so wrong the model sees garbage
3. Model silently failing and returning defaults

**DCT Tool**: Falls back to resized 224×224 crops which destroys the 8×8 JPEG grid pattern. The scoring function is extremely conservative and requires z_score > 3.0 AND peak_ratio > 0.88 to produce meaningful scores.

**UnivFD Tool**: If probe weights fail to load (`_weights_loaded_ok = False`), returns score=0.0. But even when loaded, the near-constant output suggests either label inversion (Fix #1) or preprocessing issues.

**Corneal Tool**: Requires catchlights in both eyes — very common to fail on real images.

### 🟡 BUG #3: Resize Destroying Forensic Evidence

**Evidence**: "DCT: Using resized 224×224 crop — results may be unreliable" appears on almost every image.

Resizing destroys the frequency-domain artifacts that DCT and FreqNet depend on. The interpolation overwrites GAN/diffusion artifacts with interpolation artifacts.

### 🟡 BUG #4: Ensemble Logic Over-Penalizes Real Images

**Location**: `utils/ensemble.py:419-431`

When 2+ GPU specialists score in [0.35, 0.55], mean is boosted by 1.10x as manipulation evidence. Combined with label inversion, this makes things worse.

### 🟡 BUG #5: GPU Coverage Degradation Boost

**Location**: `utils/ensemble.py:436-442`

When tools abstain, fake_score is boosted upward. Missing evidence treated as suspicion.

### 🟡 BUG #6: Verdict Threshold Too Aggressive

**Location**: `utils/thresholds.py:56`

`ENSEMBLE_REAL_THRESHOLD = 0.50` — Combined with boosting mechanisms, catches too many real images.

for detector in [run_dct, run_univfd, run_xception, run_sbi, run_freqnet]:

```python
def get_native_crop(image, size=224):
    w, h = image.size
    if w < size or h < size:
        return image.resize((size, size))  # only resize if too small
    left = (w - size) // 2
    top = (h - size) // 2
    return image.crop((left, top, left + size, top + size))
```
